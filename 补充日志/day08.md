## 核心设计
- DFA + Trie 前缀树:共享前缀只存一份(赌博/赌场 共享"赌"节点),end 标记完整词结尾
- 检测 findFirst 双层循环:外层挪起点 i,内层从 i 沿树走;走不通 break 换起点,命中 end 立即返回
- 复杂度 O(文本长度),与词库大小 N 无关(对比朴素 N 次 contains 的 O(N×M))
- Trie 启动时 @PostConstruct 从 DB build 进内存,检测全走内存不查库
- volatile trie + reload 整树原子替换 → 支持热更新,检测中的请求不受影响
- 大小写不敏感:build 和检测都 toLowerCase;返回时用原文 substring 保留大小写
- 命中抛 SENSITIVE_WORD_DETECTED(60201)

## 链路位置
扣配额 → 查模板 → 渲染 → 【敏感词 check】→ 循环投 MQ
命中即抛异常,一条 MQ 都不投(端到端已验证:控制台无"消息入队成功")

## 性能测试结果
- 1000字文本(最坏情况:词埋末尾)+ 1000 词库,预热 10000 次后平均:0.2492 ms
- 远低于目标 1ms;真实短信(几十字)更快
- 偏高原因:最坏情况全程扫描 + 每次 toLowerCase 整段复制字符串

## 关键方法论
- 单测全绿 ≠ 端到端通:单测直接 addWord("赌博")不碰 DB,覆盖不到"DB 数据是否正确建立"。
  涉及外部数据(DB)的逻辑,必须单独做端到端验证。
- 乱码区分真假:英文正常+中文乱 = 写入时坏(真乱码,要返工);中英文一起乱 = 显示编码问题(假乱码,改客户端编码即可)。
- 导中文数据最稳:用 UTF-8 的 .sql 文件 + source,不要终端粘贴。
